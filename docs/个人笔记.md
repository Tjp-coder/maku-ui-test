## cc用法

文件加载机制：CLAUDE.md 自动加载，其他文件按需触发

CLAUDE.md 的核心作用：项目入口 + 指路

**Q1：docs/ 下的文件 CC 会读吗？**

不会自动读。**两种触发方式**：

- 你在 CLAUDE.md 里写"写代码前必读 docs/架构规范.md"
- 你在对话中说"参考 docs/架构规范.md 帮我写"

**Q2：.claude/agents/ 下的 agent 文件怎么调起？**

**当前 CC 的 sub-agent 用法**：

- 在 SKILL.md 里用 `spawn_agent` 命令调起
- 或者你直接说"用 analyzer agent 处理这个需求"

**Q3：CLAUDE.md 写多长合适？**

**500-1500 字**。超过这个数 = AI 上下文被占，效果反而差。



## 定位烧成本

方案 A：纯 AI 定位

- 每次跑：15 次 × 1500 tokens（千问 VL）≈ 1-2 毛钱
- **跑 100 次回归：10-20 元**
- 跑 1000 次：100-200 元

方案 B：用 MCP 探索一次，写死 CSS

- 一次性探索：5000 tokens（CC Sonnet）≈ 5 毛钱
- 之后每次跑：**0**
- **跑 100 次回归：5 毛钱**
- 跑 1000 次：5 毛钱

差异不是"少一点点"，是 200-400 倍

但前提是：**你跑了很多次**。如果总共就跑 10 次，方案 A 也就 1-2 块钱，根本不值得折腾。

```
请帮我写步骤 4：单文件 maku 登录测试。

【背景】
项目情况见 CLAUDE.md。当前推进步骤 4，

- 不要引用 .claude/skills/harness
- 不要做"未来扩展"或"优雅抽象"
- 不要写多于 1 个测试用例
- 不要加注释解释 TypeScript 基础语法

【任务】
在 tests/cases/login.simple.spec.ts 创建单文件测试，包含以下步骤：
1. 打开 http://8.138.30.161
2. 等待登录页加载完成
3. 在用户名输入框输入 "admin"
4. 在密码输入框输入 "admin"
5. 输入验证码
6.点击「登录」按钮
7. 等待跳转完成
8. 验证已经进入首页（找一个首页特有的元素，比如左侧菜单或欢迎语）

【技术栈要求】
- Midscene PlaywrightAgent
- AI 定位为主（aiTap/aiInput/aiWaitFor/aiAssert）
- 元素描述必须带容器上下文，例如"登录表单里的『用户名』输入框"，
  不要写"用户名输入框"这种模糊描述
- 不要用 page.locator() 之类的 CSS 定位
- 单测超时设 90s（登录可能慢）

【验收标准】
- 命令 `npx playwright test tests/cases/login.simple.spec.ts --headed` 能跑通
- midscene_run/report/ 里能看到完整截图报告
- 浏览器界面能看到鼠标真的在点击

写完先解释你的实现思路（不超过 100 字），然后给我代码。
```

**1. 强制 --headed 模式** 前期一定要让浏览器**可见**地跑，眼睛看着 AI 怎么操作的。出错时直接看屏幕就知道哪步挂了，比看日志快 10 倍。

**2. 第一次跑大概率会卡某一步** 最常见的失败：

- AI 找不到登录按钮 → 描述加上"页面正中央的"
- 登录后等待时间太短没跳转 → 把 `aiWaitFor` 的 timeout 加长
- 千问 VL 第一次调用慢 → 正常，第二次会快很多

## 步骤5提示词

```
请帮我做步骤 5：把单文件登录测试拆成四层架构。

【背景】
项目情况见 CLAUDE.md。
当前文件：tests/cases/login.simple.spec.ts 已通过。
目标：保持行为完全不变，重构成 components/pages/services/cases 四层。

【拆分要求】
1. tests/components/   - 通用控件原子操作（先创建空目录，登录场景可能用不到）
2. tests/pages/        - 单页面操作
   - login.page.ts：LoginPage 类，包含：
     * inputUsername(value) - 输入用户名
     * inputPassword(value) - 输入密码  
     * clickLoginButton() - 点击登录按钮
     * 每个方法 return this 支持链式调用
3. tests/services/     - 业务流程编排
   - auth.service.ts：AuthService 类，包含：
     * login(username, password) - 完整登录流程
4. tests/cases/        - 测试用例
   - 保留 login.simple.spec.ts（不动，作为参考）
   - 新建 login.spec.ts 作为分层版本

【分层铁律（必须遵守，详见 CLAUDE.md）】
- case → 只能调 service，不准出现 aiTap/aiInput/page.locator
- service → 只能调 page，不准出现 aiTap/aiInput/page.locator
- page → 只能调 Midscene API（aiTap/aiInput 等）
- AI 元素描述必须带容器上下文

【实现要点】
- LoginPage 构造函数接受 page 参数，内部创建 agent
- AuthService 构造函数接受 page 参数
- AI 元素描述全部放在 page 层
- 链式调用：page.inputUsername().inputPassword().clickLoginButton()

【验收】
- npx playwright test tests/cases/login.spec.ts 跑通
- 行为跟 login.simple.spec.ts 完全一致
- cases/login.spec.ts 里不出现 aiTap 字眼（grep 验证）
- services/auth.service.ts 里不出现 aiTap 字眼

【约束】
- 不要引入 component 层除非真的有可复用控件（先空着）
- 不要写"未来扩展接口"
- 一次只做"拆分"这一件事，不要顺手优化别的

写完先告诉我你的拆分思路（不超过 100 字），然后给代码。
```

### 跑这个 prompt 的注意事项

#### 1. CC 可能会问要不要先备份

**让它备份**。让它保留 `login.simple.spec.ts` 作为参考文件。这样跑挂了你能对比"分层版"和"单文件版"的差异，快速定位问题。

#### 2. 拆完跑测试时建议两个都跑

```bash
npx playwright test tests/cases/login.simple.spec.ts  # 验证单文件还能跑
npx playwright test tests/cases/login.spec.ts         # 验证分层版能跑
```

**两个都通过 = 分层成功**。

#### 3. 用 grep 自检（你自己运行一下）

```powershell
# 检查 case 层有没有 AI 调用（应该 0 个）
Get-ChildItem tests/cases -Recurse | Select-String "aiTap|aiInput"

# 检查 service 层有没有 AI 调用（应该 0 个）
Get-ChildItem tests/services -Recurse | Select-String "aiTap|aiInput"
```

**有命中就是违规**，让 CC 修。这个手动检查是你做"Verifier"角色的预演——你以后写的 verifier agent 就要干这个事。
