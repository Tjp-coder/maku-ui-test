# locator-pitfalls.md

记录 AI 视觉定位（Midscene aiTap/aiInput）在本项目里踩过的坑，供 analyzer/generator 在做"这个控件该用 AI 定位还是传统定位"的决策时参考。每条坑遇到一次就补一次，不是一次性写完。

## 坑1：紧凑弹层里的选项，AI 视觉点击不稳定，重试也解决不了

现象：新增用户弹窗里「所属机构」是 `el-tree-select`，展开下拉后点击第一个（唯一的）机构选项，经常点不中；加了"重新打开下拉框→按文案点选→确认已选中"的重试循环（最多3次），三次全部失败。

根因：树形下拉紧贴触发框、可点区域很小，且弹层是 teleport 到 `document.body` 的浮层（和其他隐藏浮层共用泛化 class 名）。AI 视觉定位给出的坐标本身就有误差范围，这类元素的误差容忍空间太小，属于"重试解决不了的系统性不稳定"，不是偶发点偏。

排查方法：写一次性脚本，用原生 Playwright 定位直接点击树节点验证应用本身逻辑是否正常（`page.locator('.el-tree-select__popper .el-tree-node__content').first().click()` 能稳定选中）——能稳定选中就说明问题在 AI 定位精度，不在应用。

解法：改用原生 Playwright 定位，不再走 AI 视觉点击：
```ts
await this.page.locator('.el-form-item:has(label:text-is("所属机构")) .el-select').click();
await this.page.locator('.el-tree-select__popper .el-tree-node__content').first().click();
```

适用场景：任何"展开态浮层、可点区域紧凑、选项数量少且固定"的下拉/树形选择控件。判断信号：重试循环无法把成功率提升到可接受水平。

## 坑2：连续两个 AI 动作，后一个动作有概率点偏或"执行了但没生效"

现象两种表现：
- 表现A：填完「用户名」（第1个字段），紧接着填「姓名」（第2个字段），姓名字段最终是空的，但 Midscene 自己上报 `status: finished`（没有报错）。
- 表现B：输入用户名后立刻点击「查询」，AI 实际点到了相邻的「新增」按钮，弹出了新增弹窗而不是执行查询。

根因：两种表现是同一类问题——AI 视觉定位基于某一时刻的截图计算坐标，而前一个动作执行后页面可能还在轻微变化（校验提示、聚焦态切换、输入失焦导致的工具栏布局位移等）。第二个动作如果紧跟着执行，坐标判断可能命中的是过渡态而不是稳定态。**Midscene 的 `status: finished` 不代表值真的写进了 DOM 或点击了正确目标**，它不做结果校验。

排查方法：
1. 先怀疑"内容特殊性"（比如中文字符），写隔离复现脚本反复跑几次——如果同样代码路径、同样中文内容有时成功有时失败，就排除内容因素，坐实是非确定性时序问题。
2. 对比调试脚本和真实用例的完整调用链（包括 agent 生命周期、有没有额外等待），确认没有遗漏的确定性差异。

解法：
- 表现A（普通文本输入框）：改用原生 Playwright 定位 + `.fill()`，绕开 AI 视觉点击环节：
  ```ts
  await this.page.locator('.el-form-item:has(label:text-is("姓名")) input').fill(value);
  ```
- 表现B（相邻按钮误点）：① 输入后 `page.keyboard.press('Tab')` 强制失焦，让布局在点击前先稳定；② 定位描述里加颜色/相对位置等消歧特征而不是纯文案；③ `aiWaitFor` 里同时加入反向确认（"确认没有被误触发的另一个状态"），不要只确认正向结果。

适用场景：任何"输入后紧接着点击/输入"的连续动作序列，尤其是相邻元素视觉/语义相似（新增 vs 查询）或前一动作会触发布局变化（失焦位移、校验提示出现）的场景。
