# maku-ui-test 项目面试准备

---

## 一、一句话介绍项目（开场）

> "我自己搭了一个 AI 驱动的 UI 自动化测试框架，探索把大语言模型接进传统测试流程的工程化方案。核心是参考了业内的 Harness 设计模式，把一个 AI Agent 拆成三个职责分离的子 Agent，通过状态机调度和中间文件契约串联起来。"

---

## 二、核心问题 Q&A

### Q1：你这个项目解决了什么问题？

**答：**
传统 UI 自动化有两个痛点：

1. **元素定位脆弱**：CSS 选择器或 XPath 一旦页面改版就全挂，维护成本高。
2. **用例生成慢**：写一个完整用例要手动查 DOM、写定位、写断言，耗时且机械。

我的项目探索用 AI 视觉理解来替代硬编码定位——Midscene 框架会把页面截图发给多模态视觉模型（千问 VL），模型识别"新增用户弹窗中用户名标签右侧的输入框"在哪，然后执行点击或输入。

第二个问题——AI 虽然能生成代码，但一个 Agent 同时做分析、写代码、跑测试容易乱，我引入了 Harness 模式来解决这个问题。

---

### Q2：什么是 Harness 模式？你为什么这样设计？

**答：**
Harness 是一种 Agent 调度架构。核心思路是：**一个复杂任务不要交给单个 Agent 全权处理，而是拆成多个职责单一的 Agent，通过状态机串联。**

我参考了 TesterHome 上高飞的文章，把整个"写测试"的流程拆成三个 Agent：

```
用户需求
  ↓
Analyzer Agent  → 扫描现有代码 + 制定复用策略 → 产出 _analysis.md
  ↓
Generator Agent → 读 _analysis.md + 按四层架构写代码（不读源码）
  ↓
Verifier Agent  → 架构检查 + AI描述检查 + 跑测试 → 产出 _verification.md
  ↓
状态机 JUDGE    → 判断：通过/架构违规/描述违规/运行失败 → 按分支修复
```

这样设计的理由：
- **防止 AI 越权**：Generator 只读 `_analysis.md`，不读源码，就不会乱改已有方法
- **错误分类处理**：三种失败（架构违规、AI 描述模糊、运行超时）对应三种 fix 模式，不是简单重试
- **人工确认门**：Analyzer 标记"修改"类操作时，必须用户确认才能进入实现阶段

---

### Q3：四层架构是什么？为什么要分这四层？

**答：**
```
tests/
├── components/   # 通用 UI 控件原子操作（封装 antd 组件等）
├── pages/        # 单页面操作，只能调 Midscene API
├── services/     # 跨页面业务流程编排，只能调 page 方法
└── cases/        # 测试用例 + expect 断言，只能调 service
```

分层的核心价值是**约束 AI 的生成边界**。比如：
- case 层绝对不能出现 `aiTap()` 这种 Midscene 直接调用
- page 层不能写 `expect` 断言
- service 层不能跨页面操作 DOM

Verifier Agent 的第一步就是静态扫描这些层间违规，有违规就不跑测试，直接返回 `ARCHITECTURE_VIOLATION` 让 Generator 修复。

这样做的实际效果：我跑了好几轮，AI 生成的代码每次都能通过架构检查，因为规则写死在 Prompt 里了。

---

### Q4：AI 定位元素的稳定性怎么保证？

**答：**
这是这个方向最大的挑战，我目前的做法是：

1. **四要素描述规范**：每个 AI 定位描述必须包含"容器 + 位置 + 文案 + 类型"四个要素。比如不能写"登录按钮"，要写"登录表单底部右侧的「登录」按钮"。Verifier 的第二步就是检查这个。

2. **Verifier 静态检查**：如果描述过于模糊（只有"按钮"、"输入框"这种），返回 `AI_ONLY_VIOLATION`，让 Generator 修复描述而不是修复逻辑。

3. **已知局限**：目前用的是千问 VL Plus 模型，在复杂弹窗场景下定位精度还有问题（比如弹窗有滚动时按钮会偏移）。这是模型能力的上限，不是架构问题。

---

### Q5：项目遇到的最难的问题是什么？

**答：**
有两个印象深的：

**第一个：npm 包安装损坏**。第一次跑环境，两个并发 npm install 把 `@midscene/web` 包里的 20 多个文件弄成了 `.DELETE.xxxx` 后缀，模块找不到。排查了很久才发现是并发写文件冲突，用 PowerShell 批量重命名修复的。

**第二个：模型家族参数**。Midscene 支持多种视觉模型，但配置格式不同。千问 VL 必须设置 `MIDSCENE_MODEL_FAMILY=qwen2.5-vl`，否则坐标解析器会用错，导致点击位置全部偏移。文档里这个不明显，调了很久。

### Q6：现在项目做到什么程度了？还有哪些没做？

**答：**

**已完成（步骤 1-6）：**
- 完整的环境搭建（Node + TypeScript + Playwright + Midscene + 千问 VL）
- 登录功能四层架构测试（成功登录 + 密码错误提示验证）
- Harness 基础设施：SKILL.md 状态机 + Analyzer/Generator/Verifier 三个 Agent
- 自动重试机制：运行失败最多重试 2 次，架构违规不计入重试次数

**进行中（步骤 7）：**
- 用 Harness 跑用户管理 CRUD 测试——目前卡在弹窗元素定位偏移，表单有滚动时视觉模型定位不够准确

**明确的改进方向（简历里体现"知道边界"）：**
- 混合定位策略：稳定元素用 CSS 定位，动态/模糊元素才用 AI 视觉，减少 token 消耗
- 持久化代码索引：目前每次 Analyzer 都要重新扫描全部源文件，文件多了会变慢
- 自学习 Agent：记录定位失败的 case，下次同类场景自动调整描述（识别了但因为 token 消耗大而未实现）

### Q7：和传统 UI 自动化（Selenium/Playwright）相比优势在哪？

**答：**

| 维度 | 传统 Playwright | AI 驱动（Midscene） |
|------|-----------------|---------------------|
| 元素定位 | 手写 CSS/XPath，页面改版就坏 | 用自然语言描述，适应页面微调 |
| 用例编写效率 | 需要手动查 DOM 结构 | 描述意图即可，AI 生成定位 |
| 定位稳定性 | 高（精确匹配） | 中（依赖模型能力） |
| 维护成本 | 改版后改选择器 | 改版后改描述（但描述可能自动适应） |
| token/费用 | 无 | 每次 AI 调用有成本 |

两者不是替代关系，我项目里也保留了直接用 `page.goto()` 等原生 Playwright API 的场景——导航直接 goto 比让 AI 点菜单稳定得多。

---

### Q8：这个项目对你最大的收获是什么？（软性问题）

**答：**

最大的收获不是学了某个技术，是养成了对 AI 生成内容的**批判性审查习惯**。

举个例子：AI 第一次帮我生成的方案里有个 `loginExpectFail()` 方法，我当时觉得挺合理的。后来想了一下，这个方法名暗含了"断言"语义，而我的架构原则是"断言只在 case 层"，这个方法放在 service 层就违反了分层规则。所以我推翻了，让 `login()` 做纯操作，断言留给 case 层自己写。

这种"AI 给了方案，但我要对着工程原则过一遍"的思维，才是这个项目真正教会我的东西。

## 三、可能追问的点

**"千问 VL 和 GPT-4V 有什么区别？"**
> 能力上 GPT-4V 更强，但千问 VL 是 DashScope 国内服务，不需要代理，成本更低，适合 demo 阶段。Midscene 的视觉坐标解析器需要配置 `modelFamily`，两者配置方式不同。

**"Harness 模式是你原创的吗？"**
> 不是，参考的是 TesterHome 高飞的文章，他在字节内部实践后开源了这套思路。我做的是把这个概念落地到 Claude Code + 千问 VL 的技术栈上，并根据实际跑通情况做了一些调整，比如确认门、分层违规的处理逻辑。

**"AI 自动化测试会取代传统测试工程师吗？"**
> 不会完全取代，但会改变工作内容。这个项目验证了 AI 能处理"写代码"这个环节，但架构设计、分层规范、错误分类这些判断还是需要人来定义。AI 是执行者，工程师是裁判员和架构师。

**"步骤7没跑通，项目算完成了吗？"**
> 功能完整度上步骤 7 还差最后一公里，根因是视觉模型在弹窗有滚动时定位偏移，是模型能力问题不是架构问题。但 Harness 基础设施、四层架构、自动调度、错误分类修复这些核心设计都已经跑通了，登录用例端到端是通的。这个项目本来就是探索性质，"遇到了边界、知道边界在哪"本身也是收获。

## 四、项目关键数字（显得专业）

- 代码文件：6 个核心 .ts 文件（2 page + 2 service + 2 case）
- Agent 文件：4 个 .md（SKILL.md + 3 个子 Agent）
- 已通过测试：登录成功、密码错误提示验证 2 个用例
- 重试机制：运行失败最多 2 次，违规修复不限次
- 模型：千问 VL Plus（DashScope），每次视觉调用约 0.02-0.05 元
